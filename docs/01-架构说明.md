# FreeMail 系统架构说明

## 项目定位

FreeMail 是一个基于 Cloudflare 全家桶的临时邮箱服务，支持邮件接收、发送、转发、用户管理和自动清理。面向 C 端用户提供即用即弃的临时邮箱，面向 B 端提供 API 对接能力。

---

## 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 运行时 | Cloudflare Workers | 无服务器 JavaScript 运行环境 |
| 数据库 | Cloudflare D1 (SQLite) | 邮箱、邮件元数据、用户信息存储 |
| 对象存储 | Cloudflare R2 | 完整 EML 原始邮件存储 |
| 静态资源 | Workers Assets | 托管 HTML/CSS/JS 前端文件 |
| 邮件接收 | Cloudflare Email Routing | 将域名邮件路由到 Worker 的 `email()` 处理器 |
| 邮件发送 | Resend API / Microsoft Graph API | 双通道发件，支持多账号 |
| 认证 | JWT + Cookie | 用户登录会话管理 |
| 定时任务 | Cron Trigger | 每小时自动清理过期邮件和邮箱 |

## 模块说明

### 后端模块 (src/)

| 文件 | 职责 |
|------|------|
| `server.js` | 应用入口，实现 `fetch()` / `email()` / `scheduled()` 三个 Worker 处理器 |
| `routes.js` | 路由注册、认证中间件、JWT 验证、Root Admin Token 校验 |
| `apiHandlers.js` | 所有 REST API 的业务逻辑：邮箱创建、邮件查询、发件、用户管理 |
| `authentication.js` | JWT 创建/验证、Cookie 会话管理、邮箱独立登录验证 |
| `database.js` | D1 数据库操作封装：表初始化、CRUD、用户管理 |
| `dbConnectionHelper.js` | 数据库连接验证和获取 |
| `emailParser.js` | EML 邮件解析：提取 text/html 内容、验证码识别 |
| `emailSender.js` | 发件服务：Resend API + Microsoft Graph API 双通道，支持多账号 |
| `emailForwarder.js` | 邮件转发：根据收件人前缀匹配规则转发到指定邮箱 |
| `assetManager.js` | 静态资源处理：路由到对应 HTML 页面 |
| `commonUtils.js` | 公共工具函数 |
| `mockData.js` | 演示模式的模拟数据 |

### 前端页面 (public/)

## 请求处理流程

### HTTP 请求

```
用户请求 → Cloudflare Edge
  → fetch() → 路由匹配
    → 认证中间件（公开路径跳过、Landing 匿名、Root Token、JWT Cookie）
      → apiHandlers.js 处理业务逻辑
        → D1 查询 / R2 读取 / 发件服务
          → 返回 JSON 响应
  → 无路由匹配 → assetManager 返回静态页面
```

### 邮件接收

```
外部邮件 → Cloudflare Email Routing → email()
  → 解析收件人
  → 转发检查（FORWARD_RULES 规则匹配）
  → 读取原始 EML → 解析正文和验证码
  → 写入 R2（完整 EML 保存）
  → 写入 D1（元数据：发件人、主题、预览、验证码）
```

### 定时清理 (Cron)

```
每小时触发 → scheduled()
  → 查询过期邮箱的邮件（基于 MAIL_EXPIRE_HOURS）
  → 批量删除 R2 对象
  → 批量删除 D1 邮件记录
  → 删除无邮件的过期邮箱
  → 清理 7 天前的发送记录
```

## 认证体系

| 方式 | 适用场景 | 实现 |
|------|---------|------|
| Cookie 会话 | Web 页面用户 | 登录后设置 `iding-session` Cookie，JWT 编码 |
| Root Admin Token | API 调用 | 携带 `JWT_TOKEN` 值（Bearer / X-Admin-Token / admin_token） |
| Landing 匿名 | 首页快速使用 | `LANDING_PUBLIC=true` 时，部分接口无需认证 |
| 邮箱独立登录 | 单邮箱访问 | 通过邮箱地址+密码登录，仅可访问自己的邮件 |

## 发件通道

| 通道 | 配置 | 说明 |
|------|------|------|
| Resend | `RESEND_API_KEY` | 支持多域名 key 格式 `domain=key` |
| Microsoft Graph | `MS_ACCOUNTS` | JSON 数组，多账号自动匹配发件地址 |

系统通过 `sendEmailUnified()` 函数自动选择：优先匹配发件地址对应的 MS 账号，否则回落到 Resend。

---

## 邮件转发

通过 `FORWARD_RULES` 环境变量配置，支持两种格式：

**KV 格式：**
```
prefix1=target1@example.com,prefix2=target2@example.com,*=catchall@example.com
```

**JSON 格式：**
```json
[
  {"prefix": "xms", "email": "your@gmail.com"},
  {"prefix": "*", "email": "catch-all@gmail.com"}
]
```

匹配逻辑：收件人 local_part 以 prefix 开头则转发到对应 email，`*` 为兜底规则。

## 部署配置

### 核心文件

- `wrangler.toml` — Worker 配置（绑定、环境变量、定时任务）
- `d1-init-basic.sql` — 数据库初始化脚本
- `src/server.js` — 应用入口

### 部署命令

```bash
npx wrangler deploy
```

### 必须的 Cloudflare Dashboard 配置

1. **Email Routing**：在域名 DNS 设置中启用，添加 Catch-all 规则指向 Worker
2. **自定义域名**：Workers & Pages → 选择 Worker → Settings → Domains → 添加 `mail.codewith.site`
3. **Secrets 设置**：通过 CLI 或 Dashboard 设置 `ADMIN_PASSWORD` 和 `JWT_TOKEN`
